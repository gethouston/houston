Seccion 0

Houston Internals

Guia tecnica para desarrolladores que quieren entender y contribuir al motor de Houston.

  • Houston = app desktop (Tauri + React) + motor Rust (houston-engine) + CLIs de proveedores (Codex, Claude Code) como subprocesos.
  • El motor es frontend-agnostic: no sabe de Tauri ni React. La app solo hace glue OS (file pickers, logs).
  • Toda la logica de dominio vive en engine/. El frontend habla con el motor via HTTP REST + WebSocket.
  • Esta guia cubre: backend tokio, ciclo de un turno de agente, Workflows, y ejecucion paralela.

async / await y tokio::spawn

async fn devuelve una tarea suspendible. .await espera sin bloquear el hilo. tokio::spawn lanza trabajo en background.

engine/houston-engine-server/src/main.rs
#[tokio::main]  // crea runtime multi-thread

    let app = build_router(state);
    axum::serve(listener, app).await;  // loop HTTP forever
}

// start() responde al HTTP de inmediato; el turno corre en background
tokio::spawn(async move {
    run_start(params).await;

Patron clave en Houston: el handler HTTP responde rapido; el trabajo pesado (spawn CLI, leer stdout) va en tokio::spawn.

Seccion 1 — Rust async — Slide 5

Como se coordinan las tareas (sin jerga)

Houston no es un solo programa lineal. Hay muchas cosas pasando a la vez: varios chats, stdout de CLIs, WebSockets, file watcher. Rust/tokio usa 6 piezas para que eso no sea caos:

Canal (mpsc)Un tubo de mensajes entre dos tareas. Una escribe, otra lee. Como una cola de WhatsApp entre dos personas: el CLI manda updates, el monitor los consume.
BroadcastUn altavoz: un emisor, muchos oyentes. Cuando el engine tiene un evento nuevo, lo grita y cada cliente WS decide si le importa.
ArcPuntero compartido con contador. Muchas tareas ven el mismo ServerState sin copiarlo. Ver slide siguiente para el detalle.
Mutex / RwLockCandado: solo una tarea toca algo sensible a la vez. Evita que dos turnos de chat pisen el mismo archivo.
JoinSetVarias tareas en paralelo; cuando una termina, la recoges. Leer stdout y stderr del CLI al mismo tiempo.
select!Esperar varias fuentes y reaccionar a la primera que llegue. El WS espera: mensaje saliente, ping, o input del cliente.

No necesitas memorizar los nombres. Piensa: colas, altavoz, compartir puntero, candado, paralelo, multiplexar esperas.

Seccion 1 — Rust async — Arc en detalle

Arc: compartir estado sin copiar

El problema en Rust: por defecto un valor tiene un solo dueno. Si lo pasas a una funcion, se mueve y ya no lo tienes. Si lanzas tokio::spawn, la tarea nueva necesita poseer sus datos. Pero ServerState debe vivir en el handler HTTP, en el WS, en el file watcher, en tareas en background... imposible moverlo a todos.

Arc = Atomically Reference Counted. Envuelve el dato en un puntero con un contador de "cuantos lo estan usando".

flowchart LR subgraph heap ["Memoria heap — un solo ServerState"] State["ServerState\nDB, eventos, sesiones"] end H1["Handler POST /sessions"] H2["Handler WS"] H3["tokio::spawn run_start"] H1 -->|"Arc clone (refcount++)"| State H2 -->|"Arc clone"| State H3 -->|"Arc clone"| State
  • Arc::new(estado) crea el dato una vez en el heap y pone el contador en 1.
  • state.clone() en un Arc no copia el ServerState. Solo suma 1 al contador y te da otro puntero al mismo sitio.
  • Cuando cada tarea termina, su Arc muere y el contador baja. Cuando llega a 0, Rust libera la memoria.

Arc da lectura compartida thread-safe. Si necesitas modificar algo compartido, lo combinas con Mutex o RwLock (ej. Arc<Mutex<HashMap>>).

Seccion 1 — Rust async — Arc en otros lenguajes

Arc vs Go, Java y TypeScript

En lenguajes con GC el runtime ya comparte objetos por referencia. En Rust tu eliges explicitamente Arc para compartir entre hilos de forma segura.

Go

No hay Arc: el GC mantiene vivo lo que cualquier goroutine referencia.

// Pasas un *ServerState o lo capturas en el closure
state := &ServerState{...}
go func(s *ServerState) {
    s.RunSession()  // mismo puntero, sin refcount manual
}(state)

Cuidado: si nadie mas usa state y la goroutine aun corre, el GC igual lo mantiene mientras haya referencia.

Java

Objetos en el heap + GC. Singleton o bean inyectado = un estado para todos.

@Singleton
class ServerState { Database db; EventBus bus; }

@Inject ServerState state;  // todos los controllers
executor.submit(() -> state.runSession());

Lo mas parecido a Arc manual: AtomicReference<ServerState> o contenedor DI (Spring).

TypeScript / Node

Un modulo o closure compartido. Un solo proceso Node = un objeto en memoria.

// Patron Express: crear una vez, inyectar en cada route
const state = createServerState();
app.post('/sessions', (req, res) => startSession(state, req));
setImmediate(() => runStart(state, params));

Equivalente mental: state es el singleton; no clonas, pasas la misma referencia.

Resumen: Arc<ServerState> en Houston = el mismo "singleton de runtime" que en Express inyectarias en cada route, pero con contador explicito y liberacion determinista cuando el engine apaga (sin GC).

Seccion 2 — Arquitectura

Que es un crate?

Un crate es la unidad de compilacion de Rust: un paquete de codigo con nombre, version y dependencias. Es lo mas parecido a un modulo empaquetado que puedes reutilizar o ejecutar.

No es un archivo suelto ni una carpeta cualquiera: es un proyecto Rust completo definido por su Cargo.toml (como el package.json de npm).

Dos tipos

TipoQue produceEjemplo en Houston
Library crateUna libreria (.rlib) que otros crates importanhouston-engine-core (logica de dominio)
Binary crateUn ejecutable (.exe / binario)houston-engine-server produce houston-engine

En la carpeta del repo

engine/
  houston-engine-server/   ← crate (tiene Cargo.toml + src/)
  houston-engine-core/     ← crate
  houston-terminal-manager/← crate
  ...

El workspace (monorepo) en la raiz del repo lista todos los crates y comparte versiones de dependencias con cargo build --workspace.

Crate = paquete. Comparalo con lo que ya conoces

TypeScript / npm

  • Crate ≈ paquete en packages/foo del monorepo
  • Cargo.toml ≈ package.json
  • use houston_engine_core::... ≈ import { ... } from '@houston-ai/engine-core'
  • Workspace ≈ pnpm workspace con varios @houston-ai/*
// ui/ tiene @houston-ai/chat, @houston-ai/board...
// engine/ tiene houston-engine-core, houston-terminal-manager...
// Misma idea: paquetes chicos con un rol claro

Java / Maven o Gradle

  • Crate ≈ modulo engine-core en un multi-module project
  • Cargo.toml ≈ pom.xml / build.gradle
  • Dependencia entre crates ≈ <dependency> entre modulos
  • Binary crate ≈ modulo que genera el JAR ejecutable (Spring Boot app)
// houston-engine-server depende de houston-engine-core
// como controllers dependen de service layer

Go

  • Crate ≈ modulo Go con su go.mod (o subcarpeta de un repo)
  • Un crate library ≈ package core importable
  • Binary crate ≈ package main con func main()
  • Workspace ≈ monorepo con varios modulos bajo ./engine/...
// cmd/houston-engine/main.go  → binary (como houston-engine-server)
// internal/core/              → library (como houston-engine-core)

Por que tantos crates? Separar responsabilidades: el HTTP no mezcla parsing NDJSON; el dominio no sabe de axum. Compilas y testeas cada pieza por separado. En Houston, houston-engine-server es la "app" y el resto son "librerias internas".

pub fn build_router(state: Arc<ServerState>) -> Router {
    let v1 = Router::new()

Seccion 3 — Backend tokio/axum

Como la app se entera de todo (sin refrescar)

Cuando el agente escribe en el chat, cambia un archivo o termina un workflow, la UI debe actualizarse solita. No puedes estar haciendo fetch cada segundo a ver si hay novedades.

WebSocket = un cable abierto entre la app y el engine. El engine empuja avisos cuando pasa algo ("llego un token", "cambio el board", "workflow listo para aprobar").

Analogia: grupo de WhatsApp

  • HTTP normal = mandar un mensaje y colgar. Cada vez que quieres saber algo, vuelves a llamar.
  • WebSocket = quedarte en el grupo con la llamada abierta. Te avisan cuando hay mensaje nuevo.
  • Topic = subgrupo tematico. Ej.: solo avisos del chat mission-abc, o solo del agente bookkeeping.
  • Firehose ("*") = "meteme en todos los grupos a la vez". La app desktop hace esto: recibe todo y React decide que pantalla actualizar.

Firehose = manguera de bomberos: sale todo el agua (todos los eventos). Util cuando eres la app completa y necesitas enterarte de cualquier cosa.

Que pasa cuando hay mucho ruido

Varios agentes pueden escribir tokens a la vez. El engine no puede tumbar la app por saturacion.

4Spawn del subproceso — lanzar claude/codex

En criollo

Aqui Houston hace lo mismo que tu en terminal: ejecuta claude ... o codex ... como proceso hijo. Pero antes monta un tubo de mensajes (canal): el hijo escribira ahi "empece", "nuevo token", "termine", y quien llamo a spawn_session lee ese tubo por el otro extremo (rx).

La tarea pesada corre en background (tokio::spawn) para no bloquear quien espera los updates.

engine/houston-terminal-manager/src/manager.rs
// Devuelve una TUPLA (par): (receptor, handle de la tarea)
// -> (A, B) en Rust = devuelve dos cosas a la vez
pub fn spawn_session(...) -> (mpsc::UnboundedReceiver<SessionUpdate>, SessionHandle) {

    // Crea canal: tx = enviar, rx = recibir (como pipe o EventEmitter interno)
    let (tx, rx) = mpsc::unbounded_channel();

    // async move { } = bloque async que SE LLEVA las variables que usa (tx, provider...)
    let handle = tokio::spawn(async move {
        // let _ = ignora error de envio (patron interno; en beta se esta endureciendo)
        let _ = tx.send(SessionUpdate::Status(SessionStatus::Starting));
        // .await = espera a que termine dispatch (lanzar CLI y toda su I/O)
        session_dispatch::dispatch(&tx, provider, prompt, ...).await;
    });

    // Quien llamo spawn_session se queda con rx para leer updates
    (rx, SessionHandle { _task: handle })
}

5Loop de I/O — hablar con el proceso hijo

En criollo

Imagina ejecutar claude en tu terminal pero conectando 3 cables: stdin (le mandas el mensaje del usuario), stdout (el agente escribe tokens linea a linea), stderr (errores y warnings).

Houston escribe el prompt en stdin, cierra ese cable (para que el CLI sepa que no hay mas input), y pone dos lectores en paralelo en stdout y stderr hasta que el proceso termina.

// Conecta los 3 cables al subproceso (no usa la terminal visible del usuario)
cmd.stdout(Stdio::piped());
cmd.stderr(Stdio::piped());
cmd.stdin(Stdio::piped());

// mut = variable mutable (puede cambiar). ? = si falla, salir con error
let mut child = cmd.spawn()?;

// Escribe el prompt del usuario en stdin del CLI
stdin.write_all(prompt.as_bytes()).await;
drop(stdin);  // cerrar stdin = "ya no hay mas input" → el CLI arranca de verdad

// JoinSet = grupo de tareas paralelas; vas recogiendo las que terminan
let mut io_set = JoinSet::new();

io_set.spawn(async { read_stdout_events(stdout, tx, provider) });

// while let Some(x) = ... = mientras lleguen resultados, procesalos
while let Some(result) = io_set.join_next().await { ... }

child.wait().await;  // esperar codigo de salida (0 = ok, otro = error)

7Del canal interno al WebSocket de la app

En criollo

El paso 4-6 hablaban en un idioma interno (SessionUpdate). Este paso es el traductor: escucha ese tubo, y por cada mensaje hace dos cosas: (1) avisar a la app por WebSocket (HoustonEvent), (2) guardar en base de datos si hace falta.

Tambien guarda el PID del proceso (para cancelar con "Stop") y el session id del CLI (para retomar la conversacion mañana).

Que es Feed y que hace?

Feed es una variante de SessionUpdate que significa: "hay algo nuevo para el chat". El payload se llama FeedItem (engine/houston-terminal-manager/src/types.rs).

Piensa en el feed del chat como el timeline de mensajes de WhatsApp: cada burbuja, tool call, error o aviso del sistema es un FeedItem distinto.

FeedItemQue ves en la app
AssistantTextStreamingEl agente escribiendo en vivo (letra por letra)
AssistantTextBloque de texto final del asistente
ToolCall / ToolResult"Ejecuto esta herramienta" + resultado
ThinkingStreamingRazonamiento extendido (si el modelo lo muestra)
ProviderErrorTarjeta de error (rate limit, auth, quota...)
FinalResultTurno terminado (coste, duracion, tokens)
SystemMessageAviso interno ("Stopped by user", marker de workflow...)
FileChangesArchivos que el agente creo o modifico

Cuando llega SessionUpdate::Feed(item), el runner hace esto:

  1. Emit — sink.emit(HoustonEvent::FeedItem { ... item }) → el WebSocket lo empuja a React
  2. Persist — los items que no son streaming parcial se guardan en la DB (chat_feed) para que el historial sobreviva al cerrar la app
  3. Topic — el evento va al canal session:{session_key} (o firehose *)

En TypeScript: @houston-ai/chat recibe el JSON y renderiza cada feed_type. No parsea stdout del CLI; solo entiende FeedItem.

engine/houston-agents-conversations/src/session_runner.rs
pub async fn spawn_and_monitor(...) -> SessionResult {
    // Paso 4 nos dio rx = receptor del canal de updates del CLI
    let (rx, _handle) = SessionManager::spawn_session(...);

    tokio::spawn(async move {
        // Bucle infinito hasta que el canal se cierra (CLI termino)
        while let Some(update) = rx.recv().await {

            // match sobre ENUM = cada variante es un tipo de mensaje distinto
            // (como discriminated union en TypeScript)
            match update {
                // FEED = "nuevo item para el timeline del chat"
                SessionUpdate::Feed(item) => {
                    // item: FeedItem (texto, tool, error, etc.)
                    // Lo reenviamos al bus de eventos → WebSocket → UI
                    sink.emit(HoustonEvent::FeedItem { agent_path, session_key, item });
                    // Ademas: persistir en DB (salvo deltas de streaming)
                }
                SessionUpdate::SessionId(sid) => {
                    // sid = id de conversacion del CLI → guardar en .sid en disco
                    persist_sid_to_disk(sid);
                }
                SessionUpdate::Status(s) => {
                    // Starting | Running | Completed | Error
                    sink.emit(HoustonEvent::SessionStatus { status: s, ... });
                }
                SessionUpdate::ProcessPid(pid) => {
                    // pid = para matar el proceso si el usuario pulsa Stop
                    pid_map.insert(session_key, pid);
                }
                SessionUpdate::ResumeInvalid => {
                    // la sesion guardada ya no sirve → borrar .sid y empezar fresco
                    clear_sid_on_disk();
                }
            }
        }
    });
}

Seccion 5 — Workflows — Slide 35

Creacion del plan — que es y por que existe

En criollo

Un workflow no ejecuta "a ciegas". Primero Houston pide a la IA que descomponga el trabajo en pasos, como un project manager que escribe un checklist antes de asignar tareas.

Ese checklist es el plan (WorkflowPlan). El usuario lo revisa y aprueba. Solo despues el engine ejecuta paso a paso (Seccion 6).

Entrada al planner: el plan_prompt del workflow guardado, o el planPrompt inline del marker de chat. Ejemplo: "Audita el repo, arregla issues criticos, abre un PR".

Ejemplo de plan que devuelve la IA

{
  "steps": [
    { "id": "audit", "task": "Listar issues y deuda tecnica del repo", "depends_on": [] },
    { "id": "fix",   "task": "Corregir los issues criticos encontrados", "depends_on": ["audit"] },
    { "id": "pr",    "task": "Crear branch, commit y abrir pull request", "depends_on": ["fix"],
      "requires_approval": true }
  ]
}

depends_on forma un DAG (grafo aciclico): "fix" espera a "audit", "pr" espera a "fix". Pasos sin dependencias entre si pueden correr en paralelo (Seccion 6).

Campo del pasoSignificado
idIdentificador estable (para dependencias y UI)
taskInstruccion que recibira el agente al ejecutar ese paso
depends_onIds de pasos que deben terminar antes
use_worktreeSi true, el paso corre en copia git aislada (paralelo seguro)
requires_approvalPausa antes de ese paso (ej. enviar email, borrar archivos)
provider / modelOpcional: otro CLI/modelo solo para ese paso

pub async fn run_fanout(...) -> CoreResult<FanoutResult> {
    let mut join_set = JoinSet::new();
    loop {
        for step in &plan.steps {
            if deps_done(step, &run.steps) && eligible_status(...) {
                spawn_step(..., &mut join_set).await?;
                in_flight.insert(step.id.clone());
            }
        }
        if let Some(result) = join_set.join_next().await {
            finish_step(result).await?;
        }
        if all_done { break; }
    }
}

Pasos sin dependencias pendientes corren al mismo tiempo. Un paso fallido cancela sus dependientes (cancel_dependents).

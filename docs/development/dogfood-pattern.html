Houston Dogfood Loop — A Hybrid Validation Pattern for Autonomous PR Arcs

Author: broomva · Date: 2026-05-22 · For gethouston/houston · Tracks Issue #243 (RFC: bstack primitives + control-metalayer)

This document describes the validation loop a coding agent uses to confirm its changes work in the running Houston desktop app — from cargo build through UI clicks like a real user. The loop is hybrid because no single automation surface covers every assertion class: the engine HTTP API is the canonical state surface, cliclick drives the actual Tauri WKWebView the user sees, Interceptor drives Houston's vite frontend in real Chrome with full React DOM access, and vite HMR closes the iteration cycle on frontend code changes.

Why a dogfood loop matters for AI-agent PRs

Most agent-driven PRs fail in one of two ways: typechecks pass but the runtime doesn't behave, or the agent claims a change works without running it. Both are forms of CLAUDE.md §"No silent failures" — the policy that says reasoning is not validation, interaction is.

For Houston specifically the surface is unusually rich (Tauri 2 desktop + frontend-agnostic Rust engine + 17 engine crates + 11 React packages). A single PR can touch any combination of HTTP routes, WS events, file watchers, sidecars, and React queries. The agent needs a way to actually run the app it just changed, drive it like a user, and read back state — without a human in the loop.

The four surfaces

Engine HTTP API

curl -H "Authorization: Bearer $TOKEN" $BASE/v1/*. Source of truth. Every state mutation is API-addressable because the Tauri frontend is itself a client over HTTP/WS. Used for state assertions (workspaces, agents, preferences, providers, sessions).

cliclick on Tauri window

cliclick c:X,Y, cliclick t:"text", cliclick kp:tab kp:return. Macros logical-point coordinates to mouse + keyboard events on the real Tauri WKWebView. Sidesteps WKWebView's opacity to macOS System Events. Used for user-flow validation: clicking buttons, typing into inputs, submitting forms — exactly what a user does.

Interceptor on :1420

interceptor open, interceptor act eN, interceptor read, interceptor screenshot. Chrome extension that drives the same vite-served frontend the Tauri WebView loads. Full React DOM via semantic refs (eN). Used for richer assertions (specific DOM element states, network log capture, replayable session recordings) that cliclick can't observe.

vite HMR loop

Edit a frontend source or locale JSON → vite pushes hmr update within ~200ms → the running WebView re-renders. Engine state is preserved across HMR. Used for fast frontend iteration cycles; the cargo-build path is reserved for engine-side changes.

Architecture

When to use which surface

Assertion class Best surface Why

Engine code change → restart Tauri supervisor Watches engine/**/*.rs. Kills/rebuilds/restarts sidecar with exponential backoff. Re-read ~/.dev-houston/engine.json for new port after restart.

The canonical arc — start to finish

  1. Compile engine: cargo build -p houston-engine-server (debug ~2-4 min cold; ~30s warm).
  2. Launch app: HOUSTON_ENGINE_TOKEN=<known> pnpm tauri dev from app/. Tauri spawns vite on :1420 + builds app/src-tauri host (~46s warm) + spawns engine sidecar. The window opens.
  3. Discover engine port: jq -r '.port' ~/.dev-houston/engine.json (note: dev mode isolates to ~/.dev-houston/, NOT ~/.houston/). Bearer token is your known HOUSTON_ENGINE_TOKEN — sha256 will match token_hash field on disk.
  4. Probe: curl -H "Authorization: Bearer $TOKEN" http://127.0.0.1:$PORT/v1/health — expect {"status":"ok",...}. Without bearer: 401.
  5. Drive engine state: create workspaces, set preferences, query providers. All persist in the dev SQLite DB at ~/.dev-houston/db/houston.db.
  6. Drive UI flow: cliclick for clicks and typing on the Tauri window. Use screencapture -R to zoom on a region when coords are ambiguous; map pixel coordinates from the cropped image back to logical screen points.
  7. Capture evidence: screencapture -x -t png /tmp/houston-shots/<label>.png. The agent Reads the PNG back into context to verify visual changes.
  8. Iterate frontend: Edit any file under app/src/ or app/src/locales/<lang>/*.json. Vite logs [vite] (client) hmr update <paths> within ~200ms. The running WebView re-renders. Engine state survives.
  9. Iterate engine: Edit any engine/**/*.rs. Tauri's watcher kills the sidecar, rebuilds, and restarts at a NEW port. Re-read engine.json. Same token persists across restart because HOUSTON_ENGINE_TOKEN was set explicitly.
  10. Tear down: kill $(cat /tmp/houston-tauri-dev.pid) kills the top-level pnpm; supervisor cascades shutdown to engine + vite.

Concrete bash snippets the agent re-uses

Discover engine creds after each Tauri restart

PORT=$(jq -r '.port' ~/.dev-houston/engine.json)
TOKEN=$(cat /tmp/houston-dogfood-token.txt)
BASE="http://127.0.0.1:$PORT"
curl -sS -H "Authorization: Bearer $TOKEN" "$BASE/v1/health"

cliclick — click + type + submit

# Focus the Tauri window first (Houston is the macOS process name)
osascript -e 'tell application "Houston" to activate'
# Click into the chat input field — find coords by capturing the right panel
# region and mapping cropped pixel y back to logical screen y.
cliclick c:1662,1090
# Type the message
cliclick t:"My test mission"
# Submit
cliclick kp:return

Screenshot a region for finer coord finding

# Capture just the right panel (Houston's chat sidebar) at logical bounds
screencapture -x -t png -R 1480,200,470,980 /tmp/right-panel.png

Inject engine creds into a Chrome session at :1420 (Interceptor path)

# Before React mounts, set window.__HOUSTON_ENGINE__ so the SDK can connect
interceptor open "http://localhost:1420" --no-wait
interceptor eval --main "window.__HOUSTON_ENGINE__ = { baseUrl: 'http://127.0.0.1:'+$PORT, token: '$TOKEN' }; location.reload()"
interceptor wait-stable
interceptor read   # tree + text of the loaded page

Known gotchas (each cost time during validation)

SymptomCauseFix
~/.houston/engine.json missing after launch Tauri dev sets HOUSTON_HOME=~/.dev-houston for isolation Read ~/.dev-houston/engine.json instead
cargo check --workspace fails on CI tauri-build needs the engine sidecar at binaries/houston-engine-<triple> Run cargo build --release -p houston-engine-server before cargo check (this is what PR #242 wires into CI)
WKWebView accessibility returns missing value for all buttons React DOM is opaque to macOS System Events; the only buttons exposed are the traffic-light window controls Use cliclick with coords from screenshot inspection, OR drive :1420 via Interceptor
Typing lands on a button instead of the input Tab cycled focus to the wrong control (e.g. the microphone button next to the chat input) Click the input directly with coords; verify with a zoomed region capture
Engine port changes mid-session Tauri supervisor respawned the sidecar (CLAUDE.md §"Restart policy: exponential backoff 500ms → 30s cap") Re-read ~/.dev-houston/engine.json; same token persists if HOUSTON_ENGINE_TOKEN env var was set at launch
vite hmr invalidate /src/main.tsx "Could not Fast Refresh" main.tsx exports non-component values (a known limitation of @vitejs/plugin-react) Expect a full reload, not a fast refresh, when editing files that main.tsx imports
Frontend won't connect to engine when loaded from regular Chrome Houston's app/src/lib/engine.ts reads window.__HOUSTON_ENGINE__ only — no localStorage/env fallback like examples/smartbooks has Inject the global via interceptor eval --main before React mounts, OR add a fallback path (would be a separate PR — see "Chrome path" below)
Click via interceptor act e1 succeeds but UI doesn't advance The clicked control fires a handler that calls the engine (e.g. setPreference('locale','en')); the engine call fails silently because HoustonClient wasn't bootstrapped (see row above). interceptor diff reports changes: 0. Same fix as above. Until then, only state-independent UI surfaces (LanguageGate before EngineGate) are drivable from plain Chrome.
interceptor screenshot --save <path> ignores --save path and writes to app/interceptor-screenshot-<ts>.png Bug or undocumented behavior in interceptor 0.14.2; the JSON response carries the actual filePath Read the filePath from JSON output (--json) or move from $PWD/interceptor-screenshot-*.png after each call
pnpm tauri dev wrapper dies but child vite + engine survive Node child processes outlive their pnpm parent on some signal paths; engine's spawn_parent_watchdog only triggers on stdin EOF which can be delayed Before restarting pnpm tauri dev, kill $(pgrep -f "target/debug/houston-engine") AND $(lsof -t -iTCP:1420) — otherwise the new vite hits "Port 1420 already in use"

Dev-build caveat for the bridge. The Tauri dev build runs as a bare binary (target/debug/houston-app), NOT a registered .app bundle. interceptor macos apps doesn't list it because macOS treats it as a non-foreground process. Even with full Accessibility granted, macos tree --app Houston may not resolve. Workarounds: (1) run the release-built Houston (pnpm tauri build) which IS a .app; (2) target by PID via --pid <N>; or (3) stay with cliclick for dev-build automation.

What works right now without further user action

  • Engine API drive — all /v1/* endpoints with the dogfood bearer.
  • cliclick on Tauri window — coord-based click + type + keypress on the dev-built Houston (proven across the onboarding advance, New Mission button, chat input, Enter submit).
  • screencapture for visual evidence — already has the perms it needs.
  • vite HMR loop — frontend edits push within ~200ms.
  • Engine restart resilience — Tauri supervisor respawns on Rust file change; same HOUSTON_ENGINE_TOKEN survives across restart; new port re-read from ~/.dev-houston/engine.json.
  • Interceptor: app + frontmost queries — interceptor macos apps and macos frontmost work without Accessibility (system process info only).

What's gated on user action

  • Bridge accessibility tree on registered macOS apps — needs Accessibility TCC grant. Once granted: interceptor macos tree --app <App> returns AX tree; macos click eN + macos type <ref> "text" drive UI without coord-finding.
  • Bridge screenshots — needs Screen Recording TCC. Once granted: bridge can capture screens beyond Chrome's reach.
  • Chrome extension control of :1420 — extension must be loaded manually. Once loaded: interceptor open http://localhost:1420 + inject window.__HOUSTON_ENGINE__ via interceptor eval --main + drive React DOM via interceptor act eN.

Houston Tracker Integration — Architecture Spec

Slug: 2026-05-23-tracker-integration Status: Proposed V1 scope: Linear-only Forward-compat: Multi-provider Effort: ~8–10 weeks (V1)

1. Executive summary

Houston ships a first-class integration with Linear in V1, aligned to Linear's 2026 AppUser + AgentSession agent-delegation protocol. The integration lives as a dedicated Rust engine crate (engine/houston-linear) that speaks GraphQL directly to api.linear.app via the cynic codegen client, owns OAuth 2.0, ingests webhooks through the existing houston-relay Cloudflare Worker, and implements the AgentSession protocol natively. A new typed namespace .houston/trackers/linear/ stores a raw-plus-projection mirror; engine REST routes live under /v1/trackers/:provider/* with provider=linear the only value in V1.

The design is Linear-native in implementation, provider-agnostic in shape. Filesystem layout, route URLs, event variants, and the TrackerProvider typed enum are forward-compatible with adding Jira, GitHub Issues, Asana, or other trackers later — at zero meaningful cost today. The TicketProvider trait itself is deliberately not extracted until the rule of three triggers (concrete implementation #2 ships and #3 is on the immediate roadmap). Until then, Houston-Linear is a concrete crate, not the first instance of a framework.

2. Problem & scope

Problem

Houston's mission is to let non-technical users orchestrate teams of AI agents across hundreds of tasks. Linear is the dominant issue tracker for the kind of company Houston targets. Today the only Linear surface in Houston is a one-shot GraphQL mutation that creates a bug report (app/src-tauri/src/bug_report/linear_graphql.rs), plus opaque MCP tool access via the Composio integration. Neither supports structural orchestration: webhook reactivity, agent dispatch on issue assignment, status reconciliation, or alignment to Linear's AgentSession protocol.

In scope (V1)

  • OAuth 2.0 install for end users (workspace-scoped, registers Houston as a Linear OAuth app)
  • Bidirectional sync: read mirror of Initiatives, Projects, Cycles, Issues, ProjectMilestones, WorkflowStates; writeback of state changes, comments, AgentSession events
  • Linear AgentSession protocol implementation (AppUser registration, AgentSessionEvent ingress/egress within 5s response budget)
  • Webhook ingress via extended houston-relay Cloudflare Worker; HMAC verify at edge; deliver to engine over existing reverse-tunnel WebSocket
  • Polling reconciliation as backup for missed webhook deliveries
  • Routing policy: (team, label_set, project) → houston_agent_path
  • Board renders Linear issues with their own card variant (state pill, TEAM-N identifier, assignee, project chip, labels)
  • Connect-Linear dialog + Settings → Workspace → Linear UI (team binding, routing policy editor, sync status)
  • en/es/pt localization for every new user-facing string

Out of scope (V1, deferred)

  • Cross-workspace Initiative orchestration UI (V4)
  • Public Linear app directory listing (V1.5 — private per-workspace OAuth first)
  • Tracker abstraction (engine/houston-tracker-port trait crate) — waits for rule-of-three
  • Canonical cross-provider view (.houston/trackers/canonical/) — appears only with ≥2 providers and proven query need
  • Jira, GitHub Issues, Asana, Trello adapters — separate concrete crates when business need arrives

4. Architecture overview

The two access paths

Primary (new) — engine-direct

  • Engine speaks GraphQL to api.linear.app
  • Engine owns OAuth tokens (keychain)
  • Engine receives webhooks via houston-relay
  • Engine implements AgentSession protocol
  • Engine writes raw + projection to .houston/trackers/linear/
  • Deterministic, fast, observable

Secondary (existing) — Composio MCP

  • Agent CLI calls Composio MCP tools
  • Composio holds its own OAuth to Linear
  • No webhook reactivity, no AgentSession
  • Used when user has not OAuth-connected Houston-Linear
  • Long-tail ops escape hatch after OAuth connect
  • Zero engineering — unchanged from today

5. Filesystem layout

~/.houston/workspaces/<Workspace>/<Agent>/.houston/trackers/
  linear/                                       provider-named subdir (matches .houston/sessions/{anthropic,openai}/ convention)
    connection.json                             { provider: "linear", org_id, oauth_*, capabilities: [...] }
    raw/                                        provider-fidelity, as-received from GraphQL/webhooks
      issues/<linear_uuid>.json
      initiatives/<linear_uuid>.json
      projects/<linear_uuid>.json
      cycles/<linear_uuid>.json
      workflow_states/<linear_uuid>.json
      webhook_events.jsonl                      append-only event ledger (webhookId idempotency)
    issues.json                                 Houston's projection (today: thin shape over raw; future: canonical-aware)
    projects.json
    initiatives.json
    cycles.json
    agent_sessions/<id>.json                    per-session thread state
    sync_state.json                             { cursor, last_reconcile_at, last_error, in_flight }

  # FUTURE — same shape, only appears when concrete crate exists:
  # jira/    connection.json + raw/ + ...
  # github/  connection.json + raw/ + ...
  # canonical/   ← only when ≥2 providers + cross-provider query need proves it pays back

Why raw + projection. The raw layer captures whatever the provider sent, byte-for-byte parseable. The projection is what Houston's domain logic reads. Webhook delivery is at-least-once unordered (Linear); re-projecting from raw is idempotent. Schema drift in Linear's API is absorbed at the projection boundary — domain types never see a half-broken upstream change.

Today, with only Linear, the projection is mostly a thin shape over raw. The cost is roughly 2× disk for ticket data (negligible). The shape pays back the first time a webhook duplicate, retry storm, or schema change happens — both inevitable on a real Linear integration.

The capabilities array in connection.json is the canonical declaration of what this provider supports. UI gates and engine routes both consult it.

Anti-pattern to avoid. Adding Option<sprint_id>, Option<cycle_id>, Option<milestone_id> to a future TicketProvider trait method signature. Per Sandi Metz's "Wrong Abstraction" diagnostic, if the trait grows option parameters and adapters branch on which is Some, the abstraction is incorrect. Use capability traits (Sprintable, Cyclable) at extraction time instead.

9. Linear-specifics

GraphQL client

cynic codegen against Linear's introspected schema. Schema vendored at engine/houston-linear/schema/linear.graphql with a refresh script (scripts/refresh-linear-schema.sh) that re-introspects and produces a diff PR. Schema drift is then an explicit code review event, not a silent break.

OAuth 2.0 scopes (workspace install)

  • read — list initiatives, projects, cycles, issues, comments
  • write — update issue state, add comments
  • app:assignable — Houston AppUser can be assigned to issues
  • app:mentionable — Houston AppUser can be @-mentioned
  • webhook:write — register the webhook URL programmatically (or via Linear admin UI; both acceptable)

Webhook semantics

  • Verification: HMAC-SHA256 over raw request body, header Linear-Signature
  • Replay defense: Linear-Timestamp header; reject deliveries older than 5 min
  • Idempotency: webhookId + Linear-Delivery tracked in append-only raw/webhook_events.jsonl; dedupe on replay
  • Delivery guarantees: at-least-once, no ordering; engine projects events idempotently from raw
  • Retry policy (Linear-side): 3 retries at +1m / +1h / +6h, then auto-disable; polling reconciliation backstops

AgentSession protocol

  • Registration: on first OAuth install, engine calls agentSessionRegister to declare Houston as an AppUser in the connected org
  • Ingress: Linear emits AgentSessionEvent when the AppUser is delegated to (assigned, mentioned, or invited to a thread)
  • Response budget: 5 seconds from event receipt to first response event back to Linear. Engine sends a working event within the budget, then completes asynchronously
  • Egress: engine emits working, thought, action, complete, error events back to Linear as the Houston session progresses
  • Routing: on ingress, engine consults workspace's routing.json policy: (team_uuid, label_set, project_uuid) → houston_agent_path; matching agent receives a session start with issue context injected

Rate limit budgeter

Linear's quota is 3,000,000 complexity points per hour per OAuth app, with per-query caps. The engine's rate-limit module tracks a rolling token bucket and an explicit first: N on every paginated query (default first: 50 is multiplicative on child complexity). Webhook-driven mutations are budget-prioritized over polling-driven refreshes.

10. Forward-compatibility plan

The seams (locked in V1)

  1. Provider-scoped filesystem subdirs — .houston/trackers/<provider>/
  2. Provider-as-path-param REST routes — /v1/trackers/:provider/*
  3. Provider-tagged event variants — TrackerIssuesChanged { provider, ... }
  4. TrackerProvider typed enum in engine/houston-engine-protocol
  5. Capability declaration in connection.json
  6. Raw + projection layout on disk for idempotent re-projection

The trait extraction trigger (NOT in V1)

Extract engine/houston-tracker-port trait crate when all three conditions hold:

  1. Concrete tracker #2 ships to real users (not "planned" — running in production)
  2. That crate has been maintained for ≥1 month under real traffic
  3. Tracker #3 is committed to the immediate roadmap (= the third implementation that locks the trait)

Until then: per Metz / Fowler, the abstraction we'd extract today would be a Linear-shaped LCD with Option<...> pollution. Wait.

What the extraction looks like (when triggered)

  1. Diff engine/houston-linear and engine/houston-{#2} public surfaces
  2. Extract genuinely-common methods (likely: list_issues, get_issue, update_issue_state, add_comment, list_containers) into TicketProvider trait
  3. Provider-specific methods remain on the concrete crate; expose them via capability marker traits (AgentSessionAware, Sprintable, Cyclable, …) implemented optionally
  4. Engine-core's match provider {...} arms refactor to trait-object dispatch through a registry (mirrors the existing AI Provider newtype pattern in engine/houston-terminal-manager)
  5. Run-time downcast (as_any().downcast_ref::<dyn AgentSessionAware>()) or capability-flag lookup gates Linear-only code paths

12. Risks & mitigations

RiskSeverityMitigation
Linear AgentSession protocol edge cases (5s budget interpretation, "working"/"complete" ordering, what counts as "stalled") MED-HIGH Record real fixtures against a Linear sandbox org in C4 — not synthetic. Promote findings to research/entities/concept/linear-agent-session-*.md as discovered.
houston-relay Cloudflare Worker scaling under webhook retry storms (3 retries × 100+ in-flight issues × auto-disable timing) MED Dedupe at the relay edge using webhookId before queueing in Durable Object. Reject deliveries with stale Linear-Timestamp at edge.
Rate limit exhaustion (3M complexity pts/hr) under high-volume traffic MED Complexity-aware budgeter in C2. Explicit first: N on every paginated query. Webhook-driven mutations prioritized over polling.
OAuth refresh-token rotation races (Linear rotates on use) MED Mutex-guarded refresh in engine/houston-linear/src/auth.rs. Single in-flight refresh per connection.
State conflict (Linear + Houston both update same issue in same window) MED Linear is canonical for state. Houston writes only via documented routes that always include the prior updatedAt for optimistic concurrency. Webhooks reconcile asynchronously.
Linear app review timeline if/when published to Linear's app directory LOW (V1.5) Ship as private per-workspace OAuth app first. Public directory entry is V1.5.
Linear changes its GraphQL schema LOW cynic codegen fails the build on schema drift. scripts/refresh-linear-schema.sh produces an explicit diff PR for review.
Forward-compat over-investment (no tracker #2 ever arrives) LOW The six adjustments cost ~half a day. If tracker #2 never lands, the cost is paid; if it does, the cost is repaid 10× in avoided migration. Asymmetric bet.

13. Open items requiring user / Linear-org action

  1. Register a Linear OAuth app in your Linear org settings. Surface: LINEAR_CLIENT_ID, LINEAR_CLIENT_SECRET, LINEAR_WEBHOOK_SECRET. These go into app/src-tauri/.env.example + production env per Houston's secrets rule (env vars only, never literals).
  2. Confirm webhook hostname. Recommended: tunnel.gethouston.ai/linear/webhook/<tunnelId>. This matches the existing relay subdomain — the Worker extension in C11 mounts at this path.
  3. Linear sandbox org for C4 and C17 fixtures. Must be separate from production Linear.
  4. Houston-as-Linear-app branding — name, icon, OAuth consent screen copy. De-scoped for V1 (private app), but slots threaded through.

15. Glossary

AgentSession
Linear's 2026 protocol for delegating work to agents (AppUsers). Driven by AgentSessionEvent webhooks; 5-second response budget.
AppUser
A Linear workspace user that is an installed app rather than a human. Free, non-billable. Houston registers one per connected workspace.
Anti-corruption layer (ACL)
DDD pattern (Evans). A translation boundary between an external system's model and your domain model. Realized on disk in Houston as raw/ + projection.
Capability-typed adapter
Pattern where adapters declare what they can do via a capability set; callers gate code paths on capability presence rather than via lowest-common-denominator method unions.
Hexagonal architecture
Cockburn's ports-and-adapters pattern. Domain logic in the centre; ports define the contract; adapters implement provider-specific code outside the hexagon.
LCD collapse
Lowest-common-denominator collapse — when a unified abstraction shrinks as more providers are added, because only fields universal to all of them can be exposed.
Raw + projection
Two-layer state mirror: raw/ stores provider-fidelity payloads as received; projection is what Houston's domain reads. Re-projection from raw is idempotent.
Rule of three
Fowler / Metz refactoring heuristic: wait for the third concrete implementation before extracting the abstraction. Premature abstraction over two superficially-similar concepts is the canonical mistake.
TrackerProvider
Typed Rust enum (Linear in V1; future Jira, Github, Asana). Lives in engine/houston-engine-protocol. Tags every event, route, file, connection.

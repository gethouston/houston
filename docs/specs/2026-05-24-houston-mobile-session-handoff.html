2. Architecture (canonical)

Topology + the new push hop:

┌──────────────┐         ┌────────────────────────┐         ┌──────────────────┐
│ iOS / Android│         │  Relay Worker (DO)     │         │  User's Mac      │
│ Capacitor app│◄───────►│  tunnel.gethouston.ai  │◄──WSS──►│  houston-engine  │
│  WKWebView:  │  HTTPS  │  ├─ tunnel-do.ts       │         │  houston-tunnel  │
│  mobile/dist │  + WS   │  ├─ push registry KV * │         │  (dials out)     │
│  + native    │         │  └─ APNs/FCM sender *  │◄─Notify─┤  emits Notify    │
│  push/bio    │  APNs   │                        │  frame  │  on lifecycle    │
│              │◄─FCM────┤                        │         │                  │
└──────────────┘         └────────────────────────┘         └──────────────────┘
                                * = not yet built (credential-gated)

The single new hop: engine sends a typed Notify frame over the existing tunnel on a notify-worthy session lifecycle transition. The relay maps notifyKind onto APNs body-loc-key + FCM body_loc_key; the device localizes via its bundled resources. Engine owns policy + semantic event; relay is a dumb push pipe.

3. Decisions locked (D1–D6)

#DecisionWhy
D1Client = Capacitor (not PWA-harden / not Tauri-mobile / not Expo-RN)~95% reuse of the existing mobile/ bundle, real APNs/FCM, store presence. PWA push on iOS is too weak; Tauri mobile is less mature; Expo would discard @houston-ai DOM components.
D2Backend = Mac tunnel only (not Always-On, not user-selectable)Engine can never run on-device — it spawns CLI subprocesses (iOS forbids fork/exec). Mac tunnel is simplest. Trade-off accepted: push fires only while the Mac is awake.
D3Push origin = relay, driven by an engine-emitted Notify frameRelay is the always-on cloud component. Engine owns semantics + policy; relay is dumb.
D4iOS first; Android deferred until SDK provisioningHost has Xcode + CocoaPods. Android SDK absent. Do the verifiable platform right; don't commit unbuildable artifacts.
D5Branch base = fork/main (bun), not origin/main (pnpm)The broomva fork standardized on bun (the "#26 bun divergence"). Basing on upstream pnpm produced a dual-lockfile conflict; basing on fork is clean.
D6Notification localization = device-side (APNs loc-key / FCM body_loc_key)Server doesn't know the device locale and shouldn't have to track it. Both platforms ship this exact mechanism. Survives locale changes instantly, minimal payloads, removes a class of server state.

4. What's still open

Doable next (validatable with code + tests, no external resources)

The subscriber + the tunnel outbound-frame channel. Three components:

  1. Add an outbound-frame channel on TunnelClient. New public API TunnelClient::outbound_frame_sender() -> tokio::sync::mpsc::Sender<TunnelFrame>. Plumb the receiver into the WS write side in engine/houston-tunnel/src/connection.rs (and connection/dispatch.rs) so frames pushed in get written out to the relay. Unit-test the channel with a mock WS sink — drop frames in, assert serialized JSON appears on the sink.
  2. Subscriber task that consumes HoustonEvent::SessionStatus from a BroadcastEventSink receiver, calls notify::status_to_notify_kind(status), asks NotifyPolicy::decide(kind, session_key, now_ms), and (on Emit) builds a NotifyFrame + pushes it onto the channel. Lives in houston-engine-server (the binary that wires up the event sink + tunnel at startup). Unit-test with a synthetic broadcast stream — emit a few SessionStatus events, assert the right frames arrive at a mock channel.
  3. Wire it into houston-engine-server's startup. Where the BroadcastEventSink + TunnelClient are constructed, spawn the dispatcher with both.

locArgs design. Each NotifyKind needs a stable set of locArgs so the bundled .strings / strings.xml can use %@ / %1$s placeholders. Suggested starting set: NeedsYou → [agent_name]; Finished → [agent_name]; Failed → [agent_name]. Locale resources land later as part of the mobile-side i18n work, but the subscriber needs to commit to this shape now.

5. Environment caveats observed this session

  • tooling iOS simulator SPM resolution stalls repeatedly on Xcode 26.3 under any I/O contention — once due to an orphaned xcodebuild -list lock, once with no obvious cause past "Resolve Package Graph". CI on clean runners completes fine; locally on this host it stalls. Don't burn turns on local iOS sim builds — trust CI.
  • disk Disk hit 100% / 0 bytes free during the session and deadlocked every file write briefly. Caused by the host being ~460 GB / pre-existing full; the iOS toolchain just hit the ceiling. Recovered ~37 GB by clearing DerivedData + Rust target dirs across ~/broomva + ~/conductor Houston worktrees, then another ~11 GB sweeping the rest of the broomva workspace (symphony 1.6G · broomva-cli 2.5G + 1.2G · OmniVoice src-tauri 2.9G · houston linear-mirror 1.3G · others). Keep disk headroom; cargo regenerates target deterministically.
  • git Race on auto-merge while pushing follow-up commits. When auto-merge is armed and a second commit lands during the CI window, only the first one gets squash-merged; the follow-up commit needs a new branch + PR. Encountered on #32 → #40; recovered cleanly with a cherry-pick. Don't push follow-ups to an armed branch — open a new branch.
  • git Pre-commit (lint-staged) on the fork base runs rustfmt --check + bun run typecheck + check-locales. Long one-line assert_eq! calls will fail rustfmt --check — run cargo fmt -p <crate> before committing Rust, or rely on the hook + retry.
  • linear Linear MCP is currently not connected (and the authenticate tool isn't surfacing either — the connector appears to be fully removed). To restore: claude.ai → Settings → Connectors → +Add → Linear → authorize the Broomva workspace (not Stimulus). A previous run accidentally filed tickets in the Stimulus workspace; those were canceled but not hard-deleted (Linear MCP exposes delete_comment / delete_attachment / delete_status_update only — there's no delete_issue / delete_project; trash them in the Linear UI when you have a moment: Stimulus / BRO-49…54).
  • stash An unrelated WIP stash on feat/migrate-to-bun got popped during a recovery — recoverable from the reflog as fb82ba48b7ed64be1aa14d61d3905e748d1e4198 if you still want it.

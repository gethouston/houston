Houston Mobile — Capacitor Native Architecture Spec

Status: Proposed Date: 2026-05-23 Author: broomva (autonomous, bstack) Project: BRO · Houston Mobile Chunks: BRO-49 … BRO-54

One-line decision. Houston Mobile already ships as a PWA. Take it native by wrapping the existing mobile/ bundle in Capacitor (≈95% code reuse), keep the existing Mac reverse-tunnel as the only backend, and add native push driven by a new engine-emitted Notify frame over the existing tunnel.

1. The reality: this is not greenfield

Houston Mobile exists today and ships. The decision space below is not "how do we build a mobile app" — it is "is the PWA bet still right, and what does going native cost." The pieces already on disk:

LayerPathWhat
Clientmobile/React 19 + Vite 6 + TanStack Query 5 + react-router 7 + Tailwind v4 + framer-motion. Reuses @houston-ai/{chat,core,engine-client}.
Transporthouston-relay/Cloudflare Worker + Durable Object at tunnel.gethouston.ai. Reverse-tunnel broker and static PWA host (one origin → first-party cookies/WS).
Tunnelengine/houston-tunnel/Desktop engine dials outbound; mobile HTTP+WS multiplexes over that link. The Mac opens no inbound port.
Authmobile_access.rs, routes/tunnel.rsReusable QR → device-scoped SHA-256 bearer in engine_tokens. "Disconnect all phones" rotates + revokes.
Reactivityuse-engine-invalidation.tsWS event → query-invalidation, reconnect → refetch, tab-visibility + 2s active-session poll.

The tell. mobile/public/sw.js is a self-destruct service worker — it unregisters itself and wipes all caches ("replaces the old caching SW so existing users auto-clean"). They tried SW caching, it fought the multi-hop relay, and ripped it out. So today's PWA has no offline support and no web-push. That gap is the entire reason to consider going native.

2. The irreducible constraint

The engine spawns CLI subprocesses — codex, claude-code, gemini, composio — via houston-terminal-manager. iOS forbids fork/exec of arbitrary binaries; Android effectively forbids it too. Therefore:

The engine can never run on-device. Houston Mobile is always a thin client to a remote engine. This is equally true for PWA, Capacitor, Tauri-mobile, and React Native. No mobile technology changes it — so client-technology choice is purely about shell capabilities (push, install, background, store), not about where the engine runs.

Two valid remote engines exist: (a) the user's Mac via the tunnel — current, but dies when the Mac sleeps (phone gets 503); and (b) Always On (engine in Docker on a VPS — already shipped). This spec commits to (a), the Mac tunnel (see §9 for the consequence). Always On remains the future mobile-first unlock and is out of scope here.

3. Target architecture

The single new hop: when a session hits a notify-worthy state (Needs You / finished / failed), the engine sends a typed Notify frame over the existing tunnel. The relay forwards it to APNs/FCM for that tunnel's registered devices. The engine owns the policy and i18n (it already builds localized strings); the relay stays a dumb push pipe (no event semantics).

4. Why not the other paths

All four candidate paths keep @houston-ai/engine-client (it is just fetch + WebSocket, portable everywhere). They differ only in UI reuse, push quality, and store presence.

PathCode reuseNative pushStoreEffortVerdict
Harden PWA100%iOS: home-screen-only, fragileNoSPush too weak on iOS for the one feature mobile must nail
Capacitor wrap ⭐~95%Full APNs/FCMYesMChosen best effort-to-value
Tauri 2 mobile~90%Via plugin (less mature)YesM–LOn-brand (Rust) but mobile tooling thinner than Capacitor
Expo / React Nativeengine-client only; UI rebuiltBest-in-classYesL–XLEscape hatch if mobile becomes first-class (§11)

The iOS PWA wall (2026). Push works only since iOS 16.4, only if Added to Home Screen, no rich media, no silent push. No Background Sync / Periodic Sync / Background Fetch — no timeline. Install requires a manual Share → Add to Home Screen that most users never discover. Houston's #1 mobile job — "tell me when my agent finishes or needs me while I am away from my desk" — is exactly a push problem, and iOS PWA push is exactly the weak spot. Sources in §13.

Why Capacitor over Tauri-mobile. Tauri 2's mobile API is stable but the team is explicit it is not the "mobile-first" release — desktop is production-grade, mobile plugin coverage is partial. Capacitor is the 2026 consensus web→native bridge with a mature push/biometric plugin ecosystem and zero UI rewrite. We pay the Rust-consistency cost in exchange for shipping the push feature reliably.

5. The Notify frame (the net-new protocol surface)

This is the only genuinely new engineering. Everything else is wiring existing capabilities into a native shell.

Engine side [engine]

  • New variant on the houston-tunnel frame enum (engine/houston-tunnel/src/frame.rs): 
    Notify {
    kind: NotifyKind,      // NeedsYou | Finished | Failed
    title: String,         // localized (en/es/pt)
    body: String,          // localized
    session_key: String,   // deep-link target
}
  • Emit on session lifecycle transitions, reusing the Needs You / finished / failed signals already in agents::activity and the HoustonEvent fanout.
  • Engine owns policy: hard cap ≈3–5 notifications/day, dedup repeated states, and i18n (localized title/body). The relay never decides what is notify-worthy.

Relay side [relay]

  • Push-token registry: Worker KV keyed by tunnelId → { deviceToken, platform, bearerHash }.
  • New route POST /push/register (authenticated with the device bearer).
  • On a Notify frame from the desktop link → look up registered devices → send APNs (HTTP/2 + .p8 JWT) and FCM (HTTP v1).
  • CORS: bundling the PWA in Capacitor moves the app origin to capacitor://localhost (iOS) / http://localhost (Android). Engine auth is bearer-token (not cookies), so the relay must add a CORS allowlist for those origins on the /e/<tunnelId>/v1/*, /pair/*, /push/register routes and the WS upgrade.
  • Secrets via wrangler secret: APNs .p8 + key-id + team-id, FCM service-account JSON. Never literals (Houston secrets rule).

8. Risks & honest costs

  • ops Ops weight jumps. Today = one wrangler deploy. This adds Apple Developer ($99/yr) + Play ($25), two signing pipelines, a macOS CI runner, and store-review cycles. Real recurring maintenance.
  • review App Store Guideline 4.2 rejects thin website wrappers. Capacitor passes only if the app feels native: push + biometric + haptics + offline shell. The plan funds that (chunks 3–5), not a Safari-in-a-box. A pre-submission 4.2 self-review is a gate in BRO-54.
  • product Mac-asleep = no push. Direct consequence of the Mac-tunnel backend choice. Push only fires while the Mac is awake and connected. Must be stated plainly to users. Revisit Always On (§11) if users expect 24/7.
  • perf WebView scroll on mid-range Android runs ~20–30ms behind native. Mitigate hot lists if it shows; not a blocker for a companion app.

9. Secrets

Per the Houston rule (env / secret store only, never literals in committed files):

  • Relay Worker secrets (wrangler secret): APNs .p8 + key-id + team-id; FCM service-account JSON.
  • CI secrets: Apple signing cert + provisioning profile + App Store Connect API key; Play upload key + service-account JSON.

10. Verification strategy (Empirical, P11)

  • Chunk 1: bun install resolves Capacitor; bun run build produces dist/; cap sync ios succeeds; xcodebuild … -sdk iphonesimulator build compiles; app launches in the simulator and pairs through the relay — behavior identical to the PWA.
  • Chunk 2: engine unit tests for notify-policy (cap + dedup) with verbatim event fixtures; wrangler tail shows Notify → send; mock device token receives payload via local wrangler dev.
  • Chunk 3: run an agent on the Mac → background the phone → receive push → tap → land in the correct chat.
  • Chunks 4–6: token not readable from the WebView inspector; biometric gates entry; TestFlight + internal-track Play builds receive real push on physical devices.

11. Open questions & escape hatches

  • Always On as default backend — the real mobile-first unlock (phone independent of the Mac). Deferred by the Mac-tunnel decision; revisit when users demand 24/7.
  • Android SDK provisioning — gating the Android platform. Generate + commit android/ once the SDK is on the build host (sub-task of BRO-49).
  • Expo / React Native — if the WebView UX ceiling is hit and mobile becomes a primary surface, rebuild the UI in RN primitives. engine-client ports unchanged; only the @houston-ai DOM components are rewritten.
  • Upstream posture — this lands on the broomva fork first (bun toolchain); the upstream PR is offered to gethouston for review (and would be re-based onto their pnpm toolchain), not self-merged.

12. Decision log

  • D1 Client = Capacitor (not PWA-harden, not Tauri-mobile, not Expo). Rationale: §4.
  • D2 Backend = Mac tunnel only (not Always On, not user-selectable). Rationale: simplest, zero hosting; accepts the Mac-asleep limit.
  • D3 Push origin = relay, driven by an engine-emitted Notify frame. Rationale: relay is the always-on cloud component; engine owns semantics + i18n.
  • D4 iOS first (host has Xcode + CocoaPods); Android deferred to SDK provisioning. Rationale: do the verifiable platform right, do not commit unbuildable artifacts.
  • D5 Branch base = fork/main (bun), not origin/main (pnpm). Rationale: the fork is the primary landing target and is bun-native; basing on the pnpm upstream produced a dual-lockfile conflict.

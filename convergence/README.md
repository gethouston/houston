# Single-engine convergence ‚Äî program plan

One engine for local (desktop) + cloud (multi-tenant SaaS). Zero drift. Rust `engine/` retired at the end.

**Architecture: one host, not two addons.** Not "shared library + local supervisor + cloud control plane" ‚Äî that still lets handlers drift. Instead: ONE deployment-agnostic server (the **Houston host**, `packages/host`) with ONE router, ONE `authorize()` seam, ONE domain layer ‚Äî and two **adapter profiles** wired in `main()`:

| Port | Local adapter | Cloud adapter |
|---|---|---|
| RuntimeLauncher | subprocess spawn/sleep per agent (server mode) | CloudRun per-turn / GKE pods (exists) |
| Vfs (agent files) | real FS | GCS prefix (exists: `turn/files.ts`) |
| Identity | single-user, always-owner, offline | Supabase JWT (exists) |
| Events | FS watcher ‚Üí HoustonEvent | post-turn-sync synthetic events |
| Store (tenancy index) | FS scan of `~/.houston/workspaces` | Postgres (exists) |
| Bus | in-process | Redis (exists) |
| Scheduler | in-process cron loop | leader-elected via bus lock |
| Credentials | host-owned connect-once, `/sandbox/credential` | same code (exists) |

Drift prevention: domain logic exists once in the host; adapters pass shared **contract-test suites**; the automated cross-profile guard is `packages/host/src/dual-profile.test.ts` (boots BOTH profiles over real HTTP and asserts byte-identical v3 behavior) layered on the per-port contract suites ‚Äî the live, real-provider cross-profile run remains the MANUAL `convergence/parity-checklist.md` Layer 3 until a non-interactive test-provider credential exists (no Playwright); `/v1/capabilities` replaces "am I web/desktop" branches. These checks now run in CI on every PR: `.github/workflows/ci.yml` (`on: pull_request`) installs the pnpm workspace + the out-of-workspace Bun services and runs Biome (`pnpm check`), the workspace + Bun-service typechecks, and the `bun test` suites ‚Äî previously they ran only locally. The open/closed repo split (open local stack "Houston" vs closed cloud adapters "Houston Cloud") is documented in `BOUNDARY.md` (repo root) and machine-enforced by `scripts/check-boundaries.mjs` (`pnpm check:boundaries`, wired into the CI gate): open packages (`protocol`/`domain`/`runtime`/`ui`) may never import a cloud adapter, and inside the host only `main.ts`/`admin/**` may touch a concrete cloud adapter ‚Äî everything else goes through ports. The physical `packages/host-cloud` extraction + the cross-repo fork are a later wave; the seam is locked now so it can't rot.

**Base runtime** (`packages/runtime`) stays single-workspace/single-credential/tenancy-free forever. Gains only: skills loading (pi-native), context files, conversation rename/delete, title summarize.

**Protocol v3** = v2 conversation core verbatim, nested under `/v1/agents/:id/conversations/*`, + domain families (workspaces, agents, files, skills, routines, activities, config, providers, preferences, portable, store) + global `/v1/events` channel carrying the existing HoustonEvent vocabulary (keeps `use-agent-invalidation.ts`). Types in `packages/protocol`. Frontend talks ONLY to the host, every deployment. `@houston-ai/engine-client` keeps its function surface, transport rewritten. `packages/web/src/engine-adapter/` (the v1-faking synthesizer) gets deleted.

## Self-host (single-user VPS)

The local profile IS a server (`buildLocalHost` = `createControlPlaneServer` +
local adapters), so self-hosting is a deployment of it, not new code: bind
`0.0.0.0` (`HOUSTON_HOST_BIND`, default loopback so the desktop stays
unexposed), persist `~/.houston` to a volume, put TLS in front. One container
runs the host, which spawns pi runtimes in-container over loopback exactly like
the desktop ‚Äî "local = the cloud shrunk to one machine," literally the same
code. Artifacts + guide: `selfhost/` (Dockerfile, Caddy TLS compose, README).
Replaces the v1 `always-on/` Rust-engine image. The engine boot was verified on
a clean-room per-package install; the image build itself wants a Docker host.

## Host-sidecar release CI

The Rust `release.yml` pipeline is untouched ‚Äî it still builds the Rust engine on
a `v*` tag. The host-sidecar builds live in a SEPARATE, committed workflow,
`.github/workflows/host-sidecar-release.yml`, so they can never perturb the
releasable Rust path. It fires only off the critical path: a manual
`workflow_dispatch` or a `host-v*` tag push (NOT `v*`), producing a draft release
tagged `host-v<version>` (distinct namespace, no artifact collision). Released CI
workflow committed; push needs workflow-scoped auth / the GitHub UI (the local
`gh` token lacks the `workflow` scope to push `.github/workflows/*`).

Three platform jobs, all bun-compiling the host via the one parameterized
`scripts/build-host-sidecar.sh <triple>` (no second compile path to drift):

- **macOS** ‚Äî universal (arm64 + x86_64, lipo'd from two per-arch bun-compiles),
  signed + notarized + stapled with the SAME Apple secrets + steps as
  `release.yml::build-macos` (DMG re-sign, notarize-with-retry, full signing-chain
  verification, sidecar Developer-ID + hardened-runtime asserts), uploaded.
- **Windows** ‚Äî MSI per arch (x86_64 + aarch64 matrix), updater-key signed (`.sig`
  emitted), uploaded. None of `release.yml`'s codex/composio/git-bash CLI staging:
  pi runs providers in-process, so the host carries no CLIs.
- **Linux** ‚Äî the platform `release.yml` never builds (its `tauri.conf.json`
  `bundle.targets` are app/dmg/msi only). The CLI-free host makes a Linux DESKTOP
  build viable for the first time: x86_64 AppImage + `.deb` via tauri-action's
  `--bundles deb,appimage` (passed on the CLI, so the default config stays
  untouched), uploaded. The job also boots the bare host binary (`--verify` ‚Üí
  curls `/v1/capabilities`), which is the same binary the `selfhost/` server runs.

`build.rs::stage_host_sidecar` (the `host-sidecar` cargo feature) stages the
compiled host at the SAME `binaries/houston-engine-<triple>` externalBin name the
Rust engine uses, so `tauri.conf.json` needs no per-feature branch. End-to-end
packaged-`.app` run verification is the one manual step ‚Äî see
`convergence/packaged-app-launch.md`.

## Standing decisions

- Gemini: **dropped at cutover** (pi has google provider but API-key-only; no Google OAuth; non-technical users don't paste keys). Revisit if pi grows Google OAuth.
- OpenCode Zen + Go: **added as API-key providers** (`opencode` + `opencode-go`, two separate connections). pi ships both as built-in OpenAI-compatible gateways, so no provider engine is needed ‚Äî only an API-key credential path threaded through the stack: pi `AuthStorage` `{type:"api_key"}` creds; the host `WorkspaceCredential` gains `kind` (api-key = no refresh, `expiresAt:0` sentinel ‚Äî no Postgres migration, Pg derives `kind` on read); `refreshCredential`/`isExpiring` no-op for api-key; new `POST /agents/:id/credential/api-key` + `RuntimeChannel.saveApiKeyCredential` (ProxyChannel stores centrally AND pushes to the standing runtime for instant status; TurnChannel stores centrally, served per-turn); the per-turn `syncServedCredential` now syncs ALL providers and the cloudrun dispatch serves the agent's active provider. **API-key only** (no OpenCode OAuth) ‚Äî the connect dialog has a "Get your API key" button opening `opencode.ai/auth`. **New-engine only**: the catalog gates these behind `newEngineActive()` (the adapter sets `window.__HOUSTON_NEW_ENGINE__`), so the legacy Rust build never shows them. Cloud-eligible (it's OpenCode's own ToS-clean gateway, unlike Anthropic).
- Composio: **kept, re-wired** (reversed the earlier "cut"). pi has no MCP, but its one extension point ‚Äî custom tools ‚Äî is enough: a generic `integration_search`/`integration_execute` pi tool calls Composio's hosted v3 REST API in-process. **No bundled CLI** (the old per-arch binary is gone). Credential model = each user's OWN free "Composio for you" account (per-user `x-user-api-key`), NOT a Houston platform key; the user's key is held host-side and execution is proxied through the host (`/sandbox/integrations`), so the long-lived key never sits in the agent runtime. Behind an `IntegrationProvider` **port** (`packages/host/src/integrations/`) so a second provider slots in later. **All deployments** (capability flag, not a fork ‚Üí no drift). Build phases: (a) port + Composio adapter + registry + fake **[done ‚Äî `integrations/`, tests]**; (b) host module ‚Äî per-user key store + `/sandbox/integrations` proxy + connect/OAuth routes, registered in both `server.ts` and `local/host.ts` **[done]**; (c) the pi tool (`integration_search`/`integration_execute`, `packages/runtime/src/session/tools/integrations.ts`) **[done]**; (d) Integrations UI (`app/src/components/tabs/integrations-tab.tsx`, per-agent tab, #110) **[done]** ‚Äî but the onboarding "try" demo (`tutorial-system-prompt.ts`) still calls the old `composio` CLI, so repointing it at the in-process tool is the one remaining (d) item (product-copy rewrite, not broken code).
- Anthropic in cloud: stays off (ToS). Local keeps Anthropic + Codex OAuth. Config asymmetry, not code fork.
- Open source: Option B (open local stack: runtime + domain + host + local adapters + app + ui; cloud adapter profile closed). Apache-2.0 + DCO. Execute AFTER P6, never during. Closed repo owns no domain types, runs open contract tests, stays adapters+infra only.

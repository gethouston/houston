# Deploy Houston on a Hostinger VPS

This guide uses the Docker Compose setup in `selfhost/`. It runs the Houston host
behind Caddy, which obtains and renews the TLS certificate for your domain. The
server is single-user. Treat `HOUSTON_HOST_TOKEN` like a password because it is
the only credential protecting your workspaces and agents.

## Deploy

1. In Hostinger, provision a VPS with the Docker or Ubuntu template. Record its
   public IPv4 address.

2. In hPanel DNS, create an `A` record for the hostname you want to use and point
   it at the VPS IPv4 address. Wait for the record to resolve before starting
   Caddy.

3. In the Hostinger VPS firewall, allow inbound TCP traffic on ports `80` and
   `443`. Do not expose port `4318` publicly. Caddy reaches that port only on the
   internal Docker network.

4. Connect to the VPS over SSH, then clone and configure Houston:

   ```sh
   git clone https://github.com/gethouston/houston.git
   cd houston/selfhost
   cp .env.example .env
   openssl rand -hex 32
   ```

5. Edit `.env`. Use the token generated by the previous command:

   ```env
   HOUSTON_DOMAIN=houston.example.com
   HOUSTON_HOST_TOKEN=<generated-token>
   COMPOSIO_API_KEY=
   ```

   `COMPOSIO_API_KEY` is optional. Leave it empty to run without app
   integrations.

6. Build and start the standard self-host stack:

   ```sh
   docker compose up -d --build
   docker compose logs -f --tail=200 host
   ```

   The compose build uses `selfhost/Dockerfile` with the repository root as its
   context and builds the final `selfhost` target. The published
   `ghcr.io/gethouston/houston-engine-pod` image is intended for managed pods,
   not this VPS setup.

## Verify

Run these commands from any machine after the certificate is ready:

```sh
curl https://houston.example.com/engine/health
curl -H "Authorization: Bearer <token>" \
  https://houston.example.com/engine/v1/capabilities
```

The health response should report `{"status":"ok"}`. Connect the web client
with `https://houston.example.com/engine` and the same token. For desktop and
other client options, see the [main self-host guide](README.md).

All Houston state lives at `/data` inside the host container. Compose persists
that path in the named `houston-data` volume, normally created as
`selfhost_houston-data`. Back up that volume with the command in the operations
section of the [main self-host guide](README.md).

## Troubleshooting

- **ACME or certificate issuance fails:** confirm the domain's `A` record points
  to the VPS, wait for DNS propagation, and confirm ports `80` and `443` are
  open. Then inspect `docker compose logs caddy`.
- **A request returns 401:** the bearer token does not match
  `HOUSTON_HOST_TOKEN`. Update the client or `.env` so they match, then restart
  the stack.
- **The integrations list is empty:** set `COMPOSIO_API_KEY` in `.env` and run
  `docker compose up -d` again. Leave it empty when integrations should remain
  disabled.
